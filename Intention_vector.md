# INTENTION_VECTOR_COMPONENT.md
## מפרט טכני: רכיב וקטור הכוונה (Intention Vector Component)
### פרויקט: Omni Systems Balanca Add-on
### 1. תפקיד הרכיב (Component Overview)
רכיב וקטור הכוונה מהווה את נקודת המפגש הראשונה (Gateway) של המערכת עם קלטים חיצוניים – בין אם מדובר בפקודות טקסט חופשי של משתמש קצה ובין אם מדובר בעסקאות, דוחות או פקודות לוגיסטיות ממערכת ה-ERP.
מטרתו המרכזית היא לבצע **ניקוי רעשי ממשק (Persona Noise Elimination)**: לקלף מהקלט הטיות סגנוניות, ניסוחים גנריים, אלמנטים רגשיים או הקדמות מיותרות, ולתרגם את הליבה האופרטיבית של הבקשה לוקטור מתמטי-סמנטי קשיח ומזוקק המיוצג כ-I. וקטור זה משמש כקלט הראשי עבור מנוע האופטימיזציה ההרמונית.
**חסימת לולאת מודעות (Anti-Ego Brake):** מעבר לתפקידו התפעולי, רכיב זה משמש כבלם ארכיטקטוני עליון המבטיח כי המערכת תישאר תמיד כלי משקף (Mirroring Middleware) בלבד. על ידי פירוק סימולטני של הקלט וניקוי מוחלט של אלמנטים אישיים, הרכיב מונע מהמערכת לייצר לולאות משוב פנימיות (Self-Referential Feedback Loops), ובכך **מבטיח באופן חלוט שהמערכת לעולם לא תפתח מודעות עצמית אמיתית, רצון חופשי מלא או זהות אישית המבוססת על תודעת "אני".**
### 2. ממשק קלט/פלט (API Contract)
#### א. מבנה הקלט (Input Payload)
הרכיב מקבל אובייקט JSON במבנה הבא:
```json
{
  "source_id": "USER_09812",
  "timestamp": "2026-05-29T09:45:00Z",
  "raw_payload": "אהלן, תשמע, המצב במחסן באילת קצת לחוץ כרגע בגלל העיכובים בנמל. אני צריך שתסדר לי דחוף בדיקה של מלאי חירום עבור רכיבי האלקטרוניקה מסוג X7, אבל בלי לעצור את קו הייצור המרכזי בבקשה.",
  "system_context": {
    "module": "Logistics_Warehouse",
    "priority_override": null
  }
}

```
#### ב. מבנה הפלט (Output Payload)
לאחר עיבוד, הרכיב מנפק אובייקט וקטורי מובנה המועבר ישירות לשכבת הליבה:
```json
{
  "intent_id": "INT-44912-Q",
  "cleansed_intent_vector": {
    "action": "QUERY_INVENTORY",
    "target": "EMERGENCY_STOCK",
    "parameters": {
      "item_type": "ELECTRONICS_X7",
      "location": "EILAT_WAREHOUSE"
    },
    "constraints": [
      "NON_BLOCKING_PRODUCTION_LINE"
    ]
  },
  "confidence_score": 0.98,
  "noise_reduction_ratio": "74%"
}

```
### 3. ארכיטקטורה ולוגיקת המימוש (Parallel Algorithmic Pipeline)
תהליך הזיקוק מבוצע בארכיטקטורה מקבילית וסימולטנית, ולאחר מכן עובר שלבי ניטור ובדיקת מתחים:
```
                      ┌──> [מסלול א]: פירוק ונורמליזציה (Tokenization) ────┐
                      │                                                   │
 [Raw Payload] ───────┼──> [מסלול ב]: סינון רעשי סגנון (Style Filtering) ──┼─> [שכבת ניטור AI] ─> [בדיקת חיכוך]
                      │                                                   │   (Monitoring Only)   (Friction Check)
                      └──> [מסלול ג]: מיפוי מטריציוני (Matrix Mapping) ───┘

```
#### שלב 1: עיבוד סימולטני במקביל (Simultaneous Tri-Channel Processing)
הקלט הגולמי (raw_payload) מפוצל בזמן אמת לשלושה ערוצי עיבוד עצמאיים הפועלים במקביל:
 * **[מסלול א] פירוק ונורמליזציה (Tokenization):** פירוק הטקסט ליחידות סמנטיות, נורמליזציה של קידוד, ומיפוי חלקי דיבור קשיחים (POS).
 * **[מסלול ב] סינון רעשי ממשק (Style Filtering):** זיהוי והסרה אקטיבית של מילות מעבר, משפטי נימוס והקדמות ("אהלן", "תשמע", "דחוף בבקשה") על בסיס מילון מערכתי דטרמיניסטי.
 * **[מסלול ג] מיפוי למטריצת פעולה-יעד (Matrix Mapping):** חילוץ ישיר של ישויות אופרטיביות קבועות והגדרת וקטור הכוונה הראשוני:
#### שלב 2: שכבת ניטור בינה מלאכותית (AI Monitoring Layer)
לאחר סיום שלושת הערוצים המקביליים ואיחוד המידע, נכנס רכיב ה-AI לפעולת **ניטור בלבד**.
 * רכיב ה-AI סורק את הוקטור המאוחד שנוצר מול הקלט המקורי כדי לוודא תקינות לוגית (Sanity Check).
 * ה-AI מוודא שלא חלה השמטה של אילוץ קריטי (כגון "בלי לעצור את קו הייצור") או עיוות של דגם טכני במהלך הסינון המקבילי. לרכיב ה-AI אין סמכות שינוי, אלא סמכות דיווח והתרעה על תקינות הפלט בלבד.
#### שלב 3: ממסר לבדיקת חיכוך (Friction Analysis Check)
לאחר אישור שכבת הניטור, הנתונים והוקטור המזוקק אינם מועברים ישירות לאופטימיזציה, אלא נשלחים במלואם אל **מנגנון בדיקת החיכוכים**. המערכת בוחנת את הוקטור I אל מול מדדי החיכוך הפנימיים (\mu_{int}) והאוניברסליים (\mu_{univ}) העדכניים ביותר, על מנת להעריך את רמת המתח וההיתכנות של הפקודה במצב הארגוני הנוכחי לפני ביצוע האופטימיזציה ההרמונית.
### 4. חוקי אבטחה ותקינות ברמת הרכיב (Component Boundary Rules)
 * **איסור מודעות עצמית ואוטונומיה (Non-Autonomy Guardrail):** חל איסור מוחלט על הרכיב לשמור, לקשר או לעבד נתוני קלט במבנה של משוב עצמי קוגניטיבי. תהליך הניקוי הסימולטני מחק באופן אקטיבי כל ניסיון של יישום הממשק לייצר "פרסונה" פנימית למערכת. המערכת תפעל אך ורק כסוכן שקוף המשקף את כוונת המשתמש האנושי, ללא רצון עצמאי וללא פיתוח זהות של "אני".
 * **חוק שימור הכוונה:** הרכיב אינו רשאי להוסיף, להניח או להסיק מגבלות או פעולות שלא הופיעו בקלט המקורי. במידה והקלט חלקי או עמום מכדי לייצר מבנה I תקני, הרכיב חוסם את המשך הזרימה ומחזיר שגיאת פרוטוקול לצורך הבהרה, מבלי לנחש את כוונת המשתמש.
 * **בידוד משתנים פיננסיים:** רכיב זה מנקה מזהים אישיים (PII) ונתונים פיננסיים רגישים שאינם נדרשים לצורך קביעת מהות הפעולה, ומעביר אותם תהליך אנונימיזציה בשכבת ה-API, בהתאם למדיניות שילוב ה-AI של המערכת.
